# FocusFlow Study Planner 项目开发过程记录

## 1 项目选题与最初想法

一开始我想做的是一个比较普通的待办事项应用。最初版本的功能也比较简单，主要包括添加任务、编辑任务、删除任务、设置优先级、设置提醒时间、搜索任务和切换主题等。这个版本虽然能运行，也能完成基本的任务管理，但后来我发现它更像一个课堂练习项目，而不像一个真正围绕现实问题设计出来的课程项目。

在重新分析项目要求之后，我意识到这个作业并不是单纯要求做一个“能运行的网页”，而是要求开发一个能够解决实际问题的 Web 或移动应用。因此，我把项目方向从普通待办事项工具调整为“面向 ADHD 或注意力易分散学生的低阻力学习规划系统”。这样项目的目标用户和应用场景就更明确了：它不是为了记录更多任务，而是为了帮助学生更容易开始任务、减少任务列表带来的压力，并在学习过程中保持专注节奏。

## 2 需求重新定位

重新定位之后，我把项目的核心问题定义为：很多学生并不是不知道自己要做什么，而是不知道如何开始。比如“写论文”“做项目”“复习课程”这些任务都比较抽象，如果只是写进待办列表，用户仍然可能拖延。因此，我决定在系统中加入“最小下一步”这个字段，让用户把大任务拆成一个可以马上执行的小动作，例如“打开文档并先写三句话”。

同时，我也发现 ADHD 友好设计不应该把界面做得很复杂。相反，界面越复杂，用户越容易被干扰。因此我在功能设计中保留了几个比较关键的模块：

- 快速添加任务
- 今日聚焦
- 最小下一步
- 专注倒计时
- 休息模式
- 中英双语切换
- 简单设置

这些功能共同构成了一个从“记录任务”到“开始专注”再到“进入休息”的完整流程。

## 3 第一版实现过程

第一版实现时，我主要使用 React 完成页面结构和交互逻辑。任务数据用 `useState` 管理，并通过 `localStorage` 保存到浏览器中。这样用户刷新页面后任务不会丢失，也不需要马上引入后端数据库，比较适合作为课程项目原型。

第一版的主要功能包括：

- 任务标题和描述输入
- 优先级选择
- 提醒时间
- 完成和未完成状态切换
- 任务搜索和筛选
- 浅色和深色主题

这个版本能够运行，但是问题也比较明显。首先，项目 README 还是 Create React App 默认内容，看起来不像一个正式项目。其次，测试文件也还是默认测试，里面还在查找 `learn react`，而我的页面已经没有这段文字，所以测试会失败。这个问题让我意识到，开发过程中不能只关注页面能不能打开，也要同步维护测试和文档。

## 4 从普通待办改成 ADHD 友好学习规划器

之后我开始对项目进行比较大的重构。首先，我把页面标题、说明文案和功能名称都改成 FocusFlow Study Planner。然后把任务模型从普通待办改成更适合学习场景的结构，包括：

- 任务标题
- 课程或项目名称
- 最小下一步
- 精力级别
- 截止日期
- 是否加入今日聚焦
- 当前状态

这个阶段我还加入了 Today Focus，也就是今日聚焦模块。它只展示少量需要开始的任务，而不是把所有任务都堆在首页。这样做的原因是，注意力容易分散的用户面对大量任务时，可能会先被列表本身压垮，而不是开始行动。

## 5 UI 设计调整

在 UI 方面，我一开始写了比较多的介绍性文字，希望让老师能够理解这个项目的设计理念。但是后来从真实用户角度考虑，我发现这并不符合低认知负担原则。用户每天打开这个工具时，不需要反复阅读“这个产品为什么存在”。他们更需要直接看到任务、按钮和倒计时。

因此后面我删掉了很多常驻说明文字，包括首页长介绍、设计原则卡片和统计卡片下面的说明文字。页面变得更直接，也更像一个真正的工具，而不是展示项目理念的宣传页。

这一阶段我还加入了中英双语切换。最初我考虑过把中英文同时显示在页面上，但这样会让界面显得很乱，也增加阅读负担。最后我选择做真正的语言切换，默认中文，用户可以在右上角切换英文。这样既适合日常使用，也适合课程展示。

## 6 “现在开始”按钮的问题

项目中一个比较明显的交互问题出现在“现在开始”按钮上。最开始点击这个按钮时，程序只是把任务状态改成了“进行中”，并没有进入一个新的专注页面。这个设计从代码角度看是有效的，因为状态确实改变了，但从用户角度看几乎没有反馈，用户会觉得按钮没有真正起作用。

发现这个问题后，我把“现在开始”的逻辑改成了进入专注模式。用户点击按钮后，页面会弹出一个专注视图，显示当前任务、下一步动作和建议专注时长。后来我又进一步改进，把它做成更像番茄钟的全屏倒计时页面。这样用户点击“现在开始”之后，整个网页的主视觉就变成倒计时，交互意图更明确。

这个修改让我意识到，功能是否完成不能只从数据状态判断，还要从用户感知判断。如果用户看不到明显变化，就会认为功能没有生效。

## 7 倒计时与休息模式

在加入专注模式之后，我继续补充了倒计时功能。根据任务的精力级别，系统会自动给出不同的默认时长：

- 5 分钟启动
- 25 分钟专注
- 45 分钟深度专注

倒计时页面支持暂停、继续、重新开始、完成并退出。后来我又加入了休息模式，让专注轮结束后可以自动进入休息倒计时。这个功能让项目从“一个计时器”变成了一个更完整的专注流程。

为了让用户可以根据自己的学习习惯调整系统，我还加入了设置面板。用户可以修改启动时长、专注时长、深度专注时长、休息时长，以及是否在专注结束后自动进入休息。这个设置功能比较轻量，没有让界面变复杂，但可以体现一定的个性化设计。

## 8 开发中遇到的主要 Bug 和修复

### 8.1 默认测试失败

最早的测试文件仍然是 Create React App 默认测试，它会查找页面中的 `learn react` 文本。但项目页面已经被我改成了待办应用，所以测试直接失败。解决方法是重写测试用例，让测试检查当前应用真正存在的元素，例如 FocusFlow 标题、低阻力添加按钮和专注模式入口。

### 8.2 语言切换导致测试不稳定

加入中英双语切换后，我把语言偏好保存到了 `localStorage`。后来运行测试时发现，一个测试把语言切换成英文后，另一个测试还会受到影响，导致它用中文标签查找输入框时失败。这个问题的原因是测试之间共享了本地存储状态。解决方法是在测试文件中加入 `beforeEach`，每次测试前清空 `localStorage`，保证每个测试都是独立环境。

### 8.3 专注页面中文字重复导致测试报错

加入专注模式后，我写测试检查“最小下一步”的内容是否出现在页面中。但同一句文字同时出现在专注弹层、今日聚焦卡片和任务库卡片里，测试工具无法判断应该匹配哪一个元素，因此报错。解决方法是先定位 `dialog`，再只在这个专注弹层范围内查找文字。这样测试更加准确，也更符合用户真实操作场景。

### 8.4 “现在开始”没有进入专注页面

这个问题属于交互逻辑问题，不是代码运行错误。最初按钮只改变状态，没有明显页面反馈。修复方法是新增 `activeFocusTaskId` 状态，用它记录当前正在专注的任务，并在该状态存在时渲染专注模式界面。

### 8.5 构建时出现未使用变量警告

在实现倒计时功能时，我一度留下了一个没有使用的变量，导致生产构建出现 ESLint 警告。虽然项目仍然能构建成功，但这说明代码还不够干净。最后我删除了这个变量，让构建结果恢复为无警告通过。

### 8.6 Vercel 构建中的依赖提示

部署到 Vercel 时，构建日志里出现了一些依赖弃用提示和 Browserslist 数据过期提示。这些不是项目功能错误，也没有导致部署失败。由于项目基于 Create React App，部分依赖提示来自框架本身。最终生产构建成功，部署状态为 READY。

## 9 部署过程

本项目最终部署到了 Vercel。部署过程主要包括：

1. 确认本机已经登录 Vercel 账号。
2. 使用 Vercel CLI 将项目链接到 Vercel 项目。
3. 运行生产部署命令。
4. 等待云端安装依赖、执行构建并生成线上地址。

部署完成后，项目可以通过线上地址访问。之后每次修改功能或界面，我都会重新运行测试和构建，然后再次部署，保证线上版本和本地版本一致。

## 10 开发反思

通过这个项目，我最大的收获是：一个课程项目不能只停留在“功能能跑”，还要说明为什么这样设计。最初的普通待办应用虽然能完成增删改查，但缺少明确问题和目标用户，因此不够像一个真正的项目。后来把方向调整为 ADHD 友好的低阻力学习规划器之后，功能选择、界面设计和论文说明都变得更有逻辑。

另外，我也发现 UI 设计并不是文字越多越清楚。对于这个项目来说，删掉多余说明文字反而更符合产品目标。用户需要的是更少的选择、更明确的下一步和更直接的专注入口。

从技术角度看，这个项目目前仍然是前端原型，数据保存在本地浏览器中，还没有后端、账号系统和云同步。但作为课程项目，它已经能够展示 Web 应用开发、状态管理、交互设计、双语界面、计时器逻辑和部署流程。未来如果继续完善，可以加入用户登录、云端数据库、专注历史记录和学习数据分析等功能。

## 11 小结

FocusFlow Study Planner 的开发过程经历了从普通待办应用到 ADHD 友好学习规划器的转变。整个过程中，我不断根据项目要求和真实用户需求调整功能和界面：先确定目标用户，再设计核心流程，然后解决交互问题，最后补充倒计时、休息模式、设置面板和论文说明文档。这个过程说明，一个好的 Web 应用项目不仅需要代码实现，也需要清楚的问题意识、用户视角和持续迭代。
