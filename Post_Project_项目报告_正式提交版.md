# 基于 ADHD 友好设计的低阻力学习规划 Web 应用研究
## ——以 FocusFlow Study Planner 为例

## 摘要

在数字化学习环境不断发展的背景下，学生越来越依赖各类任务管理工具、日历工具和效率应用来辅助安排课程、项目、论文与复习计划。然而，传统的待办事项类应用往往更重视任务记录、分类和提醒功能，却较少关注用户在真实学习情境中“如何开始做一件事”“如何维持一段时间的专注”“如何在完成一个专注阶段后平稳进入下一轮工作”等关键问题。对于存在注意力调节困难、执行功能压力较大或容易被复杂任务压倒的学生而言，通用型待办工具并不能充分满足其实际需求。

基于上述问题，本文设计并实现了一款面向学生群体的低阻力学习规划 Web 应用 FocusFlow Study Planner。该系统以“降低任务启动门槛、减少认知负担、强化最小下一步、构建专注—休息闭环”为核心设计理念，围绕任务快速录入、今日聚焦、专注倒计时、休息模式、双语切换与轻量设置等功能展开实现。项目采用 React 框架进行前端开发，利用函数式组件与 Hooks 进行状态管理，并使用 LocalStorage 完成任务与设置数据的本地持久化，以保证系统在普通浏览器环境下即可直接运行。

研究与开发结果表明，FocusFlow Study Planner 并非简单的待办事项列表，而是一种围绕学习行为启动与持续专注进行交互优化的 Web 应用。它通过将“任务拆解—任务聚焦—进入专注—结束后休息”的链条整合为统一的使用流程，使传统效率工具的“记录任务”逻辑转向“支持行动”的逻辑。该项目不仅能够较好地满足课程对 Web 应用开发实践的要求，也在教育信息化、人机交互和用户中心设计结合的层面具备一定的应用价值。

**关键词：** ADHD 友好设计；学习规划；任务启动；番茄工作法；Web 应用；用户中心设计

# Section 1：引言（Introduction）

## 1.1 项目背景

随着移动互联网和 Web 应用的普及，学习管理与时间管理工具已经成为大学生和研究生日常学习中常见的数字支持系统。学生需要同时处理课程阅读、论文写作、团队项目、课堂展示、考试复习以及个人事务等多类任务，因此对任务规划与时间管理的需求极为突出。当前市场上常见的任务管理工具，包括待办事项类应用、项目看板工具和日历类工具，虽然在任务记录、时间提醒和事项分类方面功能较为成熟，但在“帮助用户真正开始做事”这一层面仍然存在明显不足。

在实际学习过程中，学生面临的问题往往并不是“不知道自己该做什么”，而是“知道该做，却很难马上开始”。例如，一名学生可能非常清楚自己需要完成课程论文初稿，但由于任务范围太大、切入点不明确、界面上同时出现的事项过多，最终迟迟无法进入执行状态。这种现象在学习压力较大、注意力容易分散或执行功能承压较高的用户群体中更加常见。由此可见，单纯强调任务记录功能的工具，并不一定能够真正改善学习行为本身。

## 1.2 问题提出

本项目聚焦的问题是：传统学习与任务管理工具在“任务启动支持”“认知负担控制”“专注执行支持”和“节奏闭环构建”方面不足，导致学生即便拥有清晰的任务列表，也未必能够顺利从计划转化为行动。

进一步而言，该问题可以细化为以下几个方面：

第一，复杂学习任务往往过于抽象，缺少便于立即执行的切入口。用户面对“写论文”“做展示”“完成项目”之类的大任务时，往往容易停留在焦虑层面，而不是迅速进入执行层面。

第二，传统待办工具常常以列表形式平铺大量事项，造成视觉与认知上的压力。面对过长的任务列表，用户需要先筛选、比较和决策，再决定做什么，这一过程本身就会消耗大量心理资源。

第三，很多工具将“计划”与“执行”分离。用户可以在工具中写下很多任务，但并不会自然过渡到专注工作模式，从而造成“记录了很多，却没有真正开始”的使用落差。

第四，一轮专注结束后，用户常常缺乏结构化的休息与重新进入机制，导致节奏中断，进而影响后续学习效率。

## 1.3 项目目标与计划开发的应用

基于上述现实问题，本项目计划开发一款面向学生的 Web 应用 FocusFlow Study Planner，其核心目标不是做一个功能堆叠型任务系统，而是设计一个**低阻力、低认知负担、支持任务启动与持续专注的学习规划工具**。

该应用的具体目标包括：

1. 通过简化任务录入流程，降低用户输入任务时的操作门槛；
2. 通过“最小下一步”设计，将抽象任务转化为立即可执行的动作；
3. 通过“今日聚焦”机制减少同时暴露的任务数量，降低视觉压力；
4. 通过全屏专注倒计时模式帮助用户从规划快速转入执行；
5. 通过休息模式与轻量设置支持更可持续的学习节奏；
6. 通过双语界面与响应式设计增强系统的可展示性与可用性。

因此，FocusFlow Study Planner 是一款面向当代学习场景的 Web 应用，目标是解决学习任务难以启动、注意力难以维持与任务界面容易过载等现实问题，并以用户中心设计思路将产品设计与前端开发实践结合起来。

# Section 2：文献综述与理论基础（Literature Review and Theory）

## 2.1 执行功能困难与任务启动问题

在学习与工作场景中，执行功能通常被理解为个体进行计划、组织、抑制冲动、维持注意力和监控行为的一组高级认知能力。相关研究指出，注意缺陷多动障碍（ADHD）与执行功能困难之间存在密切关系，尤其体现在任务启动、计划组织、工作记忆与持续专注等方面（Willcutt et al., 2005）。对于大学生而言，即使未达到临床诊断水平，执行功能压力与学业表现之间也往往存在显著关联。换言之，任务难以开始、难以维持专注，并不是简单的“拖延”，而可能与更深层的认知与行为调节问题相关。

在数字工具设计层面，这意味着有效的学习支持系统不应仅仅把任务“列出来”，而应思考如何帮助用户跨越从“知道要做”到“真正开始做”的鸿沟。尤其是在信息化学习环境中，如果工具本身需要用户进行过多输入、分类、判断和比较，反而可能进一步加重执行负担。

## 2.2 学习支持工具与认知负担控制

认知负担理论强调，个体在信息处理过程中所能投入的心理资源有限。当界面过于复杂、选择过多或结构不清晰时，用户会在尚未开始核心任务之前消耗大量注意资源。对于学习类工具而言，这一点尤其重要。一个看似功能强大的系统，若要求用户频繁地切换视线、比较选项、理解层级结构，便有可能削弱其实际使用效果。

因此，学习支持型应用在设计时需要强调“让下一步变得明显”，而不是“让所有功能都同时可见”。减少无关信息、控制视觉层级、保留关键操作入口，能够帮助用户更快完成判断与启动过程。基于这一理论，本项目在界面设计上有意减少说明性文字，控制同时展示的高优先级事项数量，并使用较直接的按钮逻辑把用户从任务卡片带入专注模式。

## 2.3 番茄工作法与时间切片理论

番茄工作法是一种以时间切片为基础的任务执行方法，其核心思想是将工作划分为一段固定长度的专注时间与短暂的休息时间，通过节奏化安排提升行动持续性（Cirillo, 2018）。相比于完全开放式的工作安排，时间切片方法更容易帮助用户建立明确的开始点和结束点，从而降低对“我要一直做多久”的焦虑。

在学习任务场景中，番茄工作法的价值不仅在于计时本身，还在于它为行动提供了边界。对于任务启动困难的用户而言，“先专注 25 分钟”比“把这件事彻底做完”更容易接受。由此，专注计时器不应被理解为单纯的附加功能，而是一种帮助用户迈出第一步的行为支持机制。本项目吸收了这一思路，在任务管理之外加入了全屏倒计时专注模式，并辅以休息模式，使应用的逻辑从“规划任务”延伸到“支持执行”。

## 2.4 用户中心设计与包容性设计

用户中心设计强调产品开发必须以真实用户的需求、行为路径和使用障碍为起点，而不是单纯围绕开发者预设的功能清单展开。对于学习类应用而言，这意味着必须理解用户在实际情境中最痛苦的环节是什么。若用户最难的是“开始”，那么设计重点就应落在降低开始门槛；若用户最难的是“被太多任务压倒”，那么设计重点就应落在减少同时可见的信息量。

与此同时，包容性设计强调产品应尽可能照顾不同认知风格与使用偏好的用户。虽然本项目特别强调 ADHD 友好性，但它并不是一款只服务于某一特定诊断群体的工具，而是一种在更广泛范围内都具有可用性的“低阻力设计”。简单、清晰、少干扰、支持行动的界面不仅有助于执行功能困难用户，也同样有利于普通学生在高压学习环境中的使用体验。

## 2.5 文献综述小结

综上所述，现有理论与研究为本项目提供了较明确的支持逻辑：第一，任务启动困难与执行功能压力具有现实基础；第二，学习支持工具必须重视认知负担控制；第三，番茄工作法所体现的时间切片与行为边界有助于专注启动；第四，用户中心设计与包容性设计为构建低阻力工具提供了方法论基础。因此，本项目在理论上具有较清晰的合理性，其设计重点并非功能堆叠，而是围绕学习行为中的关键阻碍点进行技术干预与交互优化。

# Section 3：应用开发计划与技术问题分析（Plan / Technical Analysis）

## 3.1 技术问题分析（Analysis of the Technical Problem）

本项目要解决的技术问题，并不只是“如何开发一个待办列表”，而是“如何通过 Web 应用的交互结构帮助用户降低学习任务启动门槛，并在浏览器环境中形成从任务规划到专注执行的连续流程”。

从技术层面看，相关问题主要表现为以下几点。

首先，系统必须能够支持任务信息的快速录入，同时又不能让录入过程过于复杂。如果表单字段太少，无法支撑任务执行；如果字段太多，用户又可能在输入阶段就产生负担。因此，需要在“足够有用”和“足够简单”之间找到平衡。

其次，系统必须在界面上处理任务过载的问题。若所有任务都平等展示，用户将面临选择过多的问题，因此需要一种能够突出少量优先事项的机制，使界面从“管理列表”转向“引导行动”。

再次，系统需要把计划功能与执行功能连接起来。这意味着不能让专注计时器成为一个完全独立的页面或附属插件，而应当让用户可以直接从某个任务进入专注模式，并在模式中继续看到当前任务与最小下一步。

最后，系统还要提供一定程度的个性化设置与使用连续性，这要求数据能够在浏览器端保存，并在刷新页面或重新打开应用后维持已有状态。对于课程项目而言，引入完整后端并非唯一可行方案，因此需要在项目规模、开发成本与可运行性之间做技术取舍。

## 3.2 选定技术方案与备选技术方案（Selected Technical Solution and Alternative Technical Solutions）

### 3.2.1 选定技术方案

本项目最终选择的技术方案是：**基于 React 的单页 Web 应用（Single Page Application）**。在该方案中，系统主要运行于浏览器环境，通过 React 函数组件与 Hooks 管理应用状态，并利用 LocalStorage 保存任务与设置数据。

选择这一方案的原因在于，它既能满足课程作业对 Web application 的要求，又具有较高的开发效率、展示便利性和部署可行性。用户不需要安装原生应用，只需在浏览器中打开页面即可使用，适合作为课程项目展示与单附件报告配套提交的技术成果。

### 3.2.2 备选方案一：原生移动应用

第一种备选方案是开发原生移动应用，例如 Android 或 iOS 应用。该方案的优点在于可以更好地调用系统通知、震动提醒与移动端交互能力，更符合用户在碎片化时间中的使用习惯。然而，对于本课程项目而言，原生开发的技术成本、测试成本和跨平台适配成本相对更高。如果同时兼顾 Android 与 iOS，则项目规模会迅速扩大，不利于在有限时间内完成较完整的产品原型与研究性说明。

### 3.2.3 备选方案二：传统日历/提醒型工具

第二种备选方案是将项目设计为一个以提醒、截止日期和日程展示为核心的日历型工具。该方案实现思路较清晰，也容易从时间管理角度展开。但这种方案更适合处理“何时做”，而不擅长处理“如何开始做”。如果项目仅强调提醒功能，则很难回应任务启动困难这一核心问题，因此不适合作为本项目的最终方案。

### 3.2.4 备选方案三：复杂项目管理系统

第三种备选方案是构建带有项目分组、标签、统计、看板与多维筛选能力的综合任务管理系统。该方案看似功能全面，但会明显增加界面复杂度，与本项目强调“低认知负担”和“帮助立即行动”的定位相冲突。对于目标用户而言，过多层级和配置选项反而可能造成更多使用阻力。

## 3.3 选定方案的理由（Justification of the Selected Technical Solution）

综合比较后，React 单页 Web 应用是最适合本项目的技术路线，原因主要包括以下几点。

第一，它能够在课程项目规模下实现完整、可运行且便于展示的产品原型。相比原生移动应用，Web 应用更易开发、调试和演示。

第二，它便于快速构建交互原型与迭代界面逻辑。由于本项目的核心价值集中在用户体验和交互路径设计，而不是底层系统架构，因此 React 更能支撑这种以前端交互为核心的开发目标。

第三，它具备良好的可访问性与可部署性。用户通过浏览器即可运行应用，无需下载与安装，有利于课程展示与老师评估。

第四，本项目在当前阶段并不以多用户协作、复杂权限和云端同步为重点，因此使用 LocalStorage 作为轻量持久化方案在教学项目语境下是合理的。它既能够满足状态保存需求，又不会使项目技术架构超出课程作业范围。

## 3.4 技术方案对业务/使用流程的影响评估（Business Process Impact Assessment）

尽管本项目并非企业信息系统，但仍然可以从“学习流程优化”的角度分析技术方案对使用流程的影响。

在传统学习场景中，学生的流程往往是：意识到任务存在—记录任务—拖延—临近截止日期时被动启动。该流程中最大的断点在于“记录”与“行动”之间缺乏桥梁。

本项目通过以下方式对这一流程进行重构：

1. **任务拆解阶段**：要求用户填写“最小下一步”，把模糊的大任务转化为具体动作；
2. **任务筛选阶段**：通过 Today Focus 控制可见优先级，减少同时暴露的任务数量；
3. **执行阶段**：通过 Start now 按钮直接进入专注模式，让用户从任务卡片无缝进入行动；
4. **节奏维持阶段**：在一轮专注结束后引入休息模式，帮助用户维持可持续节奏；
5. **个性化调节阶段**：通过设置调整时长和休息行为，提高工具适配度。

因此，该技术方案改变的不只是界面形式，而是用户处理学习任务的流程逻辑。它把传统“信息管理”流程转变为“行为支持”流程，具有明确的使用价值。

## 3.5 技术方案的预期收益（Perceived Benefits of the Technical Solution）

本项目技术方案的预期收益主要体现在以下方面：

第一，帮助用户更快进入任务启动状态。通过最小下一步与专注模式，用户面对的不是抽象的大任务，而是一个更具体、更容易开始的动作。

第二，降低界面导致的认知压力。Today Focus 机制减少了用户一次性看到的高优先级事项数量，减轻了被长列表压倒的可能性。

第三，提升任务执行连续性。专注模式与休息模式相互衔接，使用户从“开始”到“做一段时间”再到“短暂恢复”形成闭环。

第四，增强项目的展示性与应用性。双语界面、可运行的 Web 应用形态和较完整的功能流程，使项目既能用于课程作业，也能作为教育场景中的产品原型。

## 3.6 技术方案中使用的技术（Technologies Leveraged in the Technical Solution）

本项目主要使用了以下技术：

1. **React**：作为前端框架，用于构建单页 Web 应用；
2. **React Hooks**：用于管理组件状态、应用行为和副作用逻辑；
3. **Create React App**：作为项目初始化与构建环境，提供基本开发脚手架；
4. **Custom CSS**：用于实现页面布局、任务卡片样式、全屏专注模式与响应式界面；
5. **LocalStorage**：用于保存任务数据、语言设置和专注时长设置，实现本地持久化；
6. **Browser Notification API**：用于在浏览器环境中提供轻量提醒能力；
7. **Testing Library**：用于对关键界面行为进行基础测试，验证主要交互流程。

这些技术共同构成了一个轻量但完整的前端实现方案，能够满足课程项目对于 Web 应用开发技术展示的要求。

## 3.7 设计流程与界面/使用考虑（Design Processes and User Interface / Utilization Considerations）

本项目在设计流程中，重点考虑了目标用户的真实使用障碍，而非单纯追求功能数量。设计考虑主要包括以下几点：

### 3.7.1 低阻力录入

任务创建只保留最必要的信息：标题、课程/项目、截止日期、精力级别、最小下一步与是否加入今日聚焦。这种设计避免了复杂字段带来的疲劳感。

### 3.7.2 降低视觉噪音

主界面尽量控制说明性文本，减少花哨装饰与冗余信息，使页面更像一个可以立即操作的工具，而不是需要阅读理解的大型仪表板。

### 3.7.3 强化下一步动作

“最小下一步”是界面逻辑中的核心设计点。它帮助用户把“大任务”变成“可立即执行的小动作”，减少开始时的不确定感。

### 3.7.4 专注模式沉浸化

进入专注模式后，界面转为大号倒计时显示，并突出当前任务与下一步提示，弱化其他界面元素，使用户注意力集中在单一目标上。

### 3.7.5 双语与展示友好性

系统支持中文与英文切换，有助于满足不同语言环境下的使用与展示需求，也增强了课程项目答辩时的表达灵活性。

## 3.8 技术方案的使用参数、使用指南与流程说明（Technical Solution Usage Parameters / Usage Guide / Schematic）

本项目的基本使用流程如下：

1. 用户打开应用主界面；
2. 点击创建任务，输入标题、课程/项目、截止日期与最小下一步；
3. 选择任务的精力级别，例如 5 分钟启动、25 分钟专注或 45 分钟深度专注；
4. 根据优先级决定是否将任务加入 Today Focus；
5. 在主界面查看今日聚焦任务；
6. 点击“Start now”进入专注模式；
7. 在全屏倒计时界面中围绕当前任务展开学习；
8. 倒计时结束后，系统进入休息模式或根据设置自动开始休息；
9. 用户可在设置中调整专注时长、休息时长与自动休息选项。

其中，主要参数包括：

- Quick Start Duration（快速启动时长）
- Standard Focus Duration（标准专注时长）
- Deep Focus Duration（深度专注时长）
- Break Duration（休息时长）
- Auto-start Break（是否自动开始休息）

该流程体现出本项目并非只是静态记录任务，而是强调从任务创建到专注执行的全过程支持。

## 3.9 利益相关者分析（Stakeholders）

本项目涉及的主要利益相关者包括：

第一，**学生用户**。这是系统的直接使用者，也是最核心的利益相关者。系统的设计目的即在于帮助其降低学习任务启动难度并改善专注体验。

第二，**教师与导师**。虽然教师不是直接操作系统的主要用户，但学生更有效的学习规划与任务执行可以间接提升课程项目推进、论文写作与学习反馈的质量。

第三，**项目开发者/作者**。作为设计与实现者，开发者通过本项目展示了前端开发能力、用户需求分析能力与教育技术场景中的产品思考能力。

第四，**教育技术应用场景中的学校支持系统**。从更广义角度看，若此类工具进一步发展，可作为高校学习支持、学生发展辅导或数字学习平台中的辅助模块。

## 3.10 经验教训（Lessons Learned）

在本项目的设计与实现过程中，可以总结出以下几点经验。

第一，学习支持工具并不是功能越多越好。与其不断增加标签、统计和复杂筛选，不如优先解决用户最痛的一个问题——开始行动。

第二，产品逻辑应服务于使用行为，而不是停留在信息管理。很多待办工具能够帮助用户“写下任务”，但不能帮助用户“真正开始”，这提醒开发者在设计时要更加关注行为路径。

第三，针对特定用户痛点的小而精设计，比面向所有人需求的大而全设计更容易形成明确价值。FocusFlow 的差异性正是来自其对“任务启动困难”这一问题的持续聚焦。

第四，课程项目中的技术选择应与目标匹配。若项目重点是交互和用户体验，那么轻量 Web 技术往往比复杂后端架构更合适。

# Section 4：研究发现与结论（Findings / Conclusions）

## 4.1 项目实现结果

经过设计与开发，FocusFlow Study Planner 已经实现了一套可运行的前端 Web 应用原型，能够支持任务创建、任务筛选、今日聚焦、进入专注模式、专注倒计时、休息模式、设置调整、双语切换与本地数据保存等核心功能。从课程项目的角度看，该系统已经具备较完整的应用形态，而非仅停留在概念稿或界面静态展示阶段。

## 4.2 项目主要发现

本项目的主要发现包括以下几个方面。

首先，任务启动困难可以通过界面与交互设计被一定程度地技术化处理。尽管数字工具无法彻底替代人的自我管理，但通过把任务拆解为“最小下一步”，确实能够在产品逻辑层面降低行动门槛。

其次，减少任务暴露数量比单纯丰富任务管理维度更符合目标用户需要。对学习压力较大的用户来说，看到更少但更重要的任务，比看到更完整的任务系统更有帮助。

再次，把专注计时器直接嵌入任务流程，而非作为独立工具存在，有助于提升从规划到执行的流畅度。用户不必在多个应用之间切换，也更容易在明确目标后立即行动。

最后，Web 应用作为课程项目载体具有较强优势。它既能展示真实的开发能力，又适合快速演示和持续迭代，尤其适合这种以交互体验为核心的项目类型。

## 4.3 项目的限制

尽管本项目已经能够实现主要功能，但仍然存在若干限制。

第一，目前数据仅保存在浏览器本地，尚未实现账号系统与跨设备同步。

第二，系统当前为单用户结构，不支持多人协作或教师—学生联动等扩展场景。

第三，提醒与通知能力依赖浏览器环境，不同浏览器可能存在行为差异。

第四，目前项目更多体现为高质量前端原型，尚未进一步扩展为带后端数据库和分析功能的完整生产级系统。

这些限制并不会否定其课程项目价值，反而清晰展示了该项目从课程原型走向实际产品时的潜在扩展路径。

## 4.4 结论

总体而言，FocusFlow Study Planner 成功回应了本课程 Post Project 作业中“围绕现实问题开发网络或移动应用”的要求。该项目从学生学习情境中的真实难点出发，将执行功能支持、时间切片思路和低认知负担设计整合进一个 Web 应用之中，不仅体现了前端开发技术的应用，也展示了围绕目标用户需求进行产品设计的能力。

与普通待办事项工具相比，本项目更强调“帮助行动”而不是“记录任务”；更关注“任务启动支持”而不是“任务堆叠管理”；更重视“专注—休息闭环”而不是单纯的时间提醒。正因为如此，该项目能够体现出较明确的研究意义与实践价值，并适合作为进阶项目课程中的 Web application 成果提交。

# Section 5：总结与建议（Summary / Recommendations）

## 5.1 总结

本文围绕 FocusFlow Study Planner 的项目背景、理论基础、技术问题、选定方案、界面设计、实现逻辑与项目价值进行了较系统的说明。项目针对学生在学习过程中普遍存在的任务难以启动、任务列表压力大、专注难以持续等现实问题，提出并实现了一种更强调行动支持的 Web 应用方案。

从开发成果看，本项目已经具备以下特点：

1. 具有明确的问题导向和目标用户；
2. 实现了较完整的 Web 应用功能流程；
3. 采用了合理且与课程规模匹配的技术栈；
4. 体现了用户中心设计和低认知负担设计思路；
5. 能够较好地支撑课程对项目研究性说明与应用实现的双重要求。

因此，本项目既是一个技术实现成果，也是一个围绕学习支持场景展开的产品设计实践。

## 5.2 建议

基于当前实现情况，后续可从以下几个方向进一步优化与扩展。

第一，增加用户账号系统与云端数据库，实现跨设备同步与长期数据保存。这样可以使系统从课程项目原型向真实可持续使用的产品演进。

第二，引入学习数据统计与历史记录功能，例如专注次数、累计专注时长、任务完成趋势等，以增强用户对自身行为模式的了解。

第三，增加个性化推荐能力，例如根据用户常用时长、任务完成情况或精力水平给出更适合的专注建议。

第四，扩展通知与提醒能力，如可选声音提醒、桌面通知优化以及更灵活的休息提示逻辑。

第五，若未来面向更大范围的教育场景应用，可考虑与课程平台、学习支持系统或学生发展辅导模块进行接口整合。

综合来看，FocusFlow Study Planner 已经作为一个课程项目较好地完成了从问题发现、理论支撑、技术设计到应用实现的全过程。其后续扩展方向明确，也说明本项目具有进一步深化的潜力。
